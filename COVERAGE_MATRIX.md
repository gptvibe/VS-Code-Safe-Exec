# Safe Exec Coverage Matrix

This matrix describes the current implementation in this repository. It is intentionally conservative: if a surface is not wired to a stable VS Code event or an explicit Safe Exec command, it is not claimed as covered.

| Surface | Current coverage | Implemented via | What Safe Exec does today | What it does not do |
| --- | --- | --- | --- | --- |
| Terminal interception | Partial | `window.onDidStartTerminalShellExecution` and `window.onDidEndTerminalShellExecution` in `src/terminalInterceptor.ts` | Matches configured terminal rules after VS Code shell integration reports a start event, attempts interrupt and dispose, asks for approval, and replays approved commands in a fresh terminal. | It is post-start, not pre-execution blocking. The original command may already have produced side effects. Terminals that do not emit usable shell-integration events are outside this flow. |
| Protected commands | Explicit only | Safe Exec proxies and `safeExec.runProtectedCommand` in `src/commandInterceptor.ts` | Shows approval only when the command is invoked through a Safe Exec proxy or wrapper command. The stable built-in coverage now includes tasks, notebook execution, interactive input, extension install or uninstall, and wrapper-first workspace or window commands such as `vscode.openFolder` and `vscode.newWindow`. | It does not transparently intercept raw built-in commands. Direct invocations, raw keybindings, or extension-to-command calls bypass this surface unless they opt into the wrapper. |
| Edit interception | Partial | `workspace.onDidChangeTextDocument` in `src/editInterceptor.ts` | Snapshots `file` and `untitled` text documents, rolls back suspicious edits, offers diff review, and reapplies approved changes. | It is post-change rollback and reapply, not pre-edit approval. Non-text mutations or filesystem changes that never surface as text-document change events are outside this surface. |
| File operations | Partial | `workspace.onWillCreateFiles`, `workspace.onWillDeleteFiles`, `workspace.onWillRenameFiles`, `workspace.onDidCreateFiles`, `workspace.onDidDeleteFiles`, and `workspace.onDidRenameFiles` in `src/fileOperationInterceptor.ts` | Observes and classifies create, delete, and rename operations when VS Code emits those events. Delete and rename flows can capture bounded snapshots and later restore supported paths. | It does not show a file-operation approval prompt or cancel the operation. Coverage is limited to VS Code event-backed paths. External disk changes and bypass paths are outside this surface. |
| Raw built-in commands | None unless wrapped | No transparent hook; only explicit Safe Exec command routing in `src/commandInterceptor.ts` | Nothing by default. Coverage exists only if the raw command is replaced with a Safe Exec proxy or wrapper call. | Safe Exec does not override arbitrary built-in commands behind the scenes. |
| Non-integrated terminals | None | No hook outside VS Code integrated terminal shell-execution events | Nothing. | External terminals, extension-provided terminals without usable shell integration, or any other shell surface that never emits `onDidStartTerminalShellExecution` are outside Safe Exec terminal coverage. |
| `workspace.fs` | Not relied on | No dedicated `workspace.fs` interception path | Nothing directly. | `workspace.fs` create, delete, rename, and write paths may bypass Safe Exec file-operation handling. Safe Exec does not claim these calls are intercepted or recoverable. |
| External disk changes | None as a file-operation surface | No OS-level filesystem watcher or sandbox boundary | Nothing directly. | Changes made by other applications, scripts, or terminals outside the covered VS Code flows are not blocked or recovered as file operations. |
| Malicious extensions | None as a security boundary | Safe Exec uses normal extension APIs and local storage | Safe Exec can only cover the explicit VS Code events and proxy commands it is wired to. | It does not sandbox other extensions. A malicious extension can call raw commands, spawn processes, use bypass paths, or tamper with local state. |
| Untrusted workspaces | Limited | Extension activation plus workspace-trust status handling in `src/extension.ts` | Safe Exec still activates, surfaces trust state, records trust changes, and can use supported APIs where VS Code allows them. | Workspace Trust is not a sandbox and does not make Safe Exec a hard security boundary. Coverage still depends on the same VS Code events and explicit command wrappers. |

## Notes

- File-operation audit events named `intercepted` mean Safe Exec preflighted a delete or rename operation before completion and created a record. They do not mean the operation was blocked.
- File-operation recovery is bounded by snapshot byte limits, per-operation file-count limits, URI support, binary snapshot settings, and restore conflict checks.
- Terminal replay is best effort. Safe Exec preserves `cwd` and replayable launch options only when VS Code exposes them, and replay can fall back to `sendText(...)`.
